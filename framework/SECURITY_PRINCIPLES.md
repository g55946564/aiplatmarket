복사해서 전달할 문구

이 프로젝트의 백엔드 API(특히 Firestore/DB 연동, 사용자 데이터 저장·조회)를 설계할 때 아래 3가지 보안 원칙을 반드시 지켜주세요. 이건 메인 AI플랫마켓 서버에서 이미 실제 취약점 2건을 고친 뒤 확정한 원칙입니다.
1. DB 문서를 통째로 API 응답에 펼치지 않기 (화이트리스트 방식)
res.json({ ...doc.data() })처럼 문서 전체를 그대로 응답에 넣지 마세요. 나중에 민감한 필드(연락처, 개인 분석결과 원본 등)가 추가되면 개발자가 의식하지 못한 채 API로 자동 노출됩니다. 항상 응답에 내보낼 필드를 명시적으로 하나씩 나열하세요.
js// ❌ 하지 말 것
res.json({ id: doc.id, ...doc.data() });

// ✅ 이렇게
const d = doc.data();
res.json({ id: doc.id, title: d.title, summary: d.summary, createdAt: d.createdAt });
2. "전체 조회"·"관리자용" 기능은 반드시 인증 확인
특정 사용자 하나가 아니라 여러 명/전체 데이터를 조회하는 API(예: ?all=true, ?userId 없이 전체 조회, 관리자 대시보드용 API)는 반드시 인증 토큰을 확인하는 미들웨어를 거치게 하세요. "본인 것만 조회"(예: userId 지정)는 비로그인도 허용 가능하지만, "여러 명 것 한번에"는 항상 관리자 인증이 있어야 합니다.
3. 새 API/컬렉션을 만들 때마다 이 질문을 스스로 하기
"로그인 안 한 사람이 URL만 알면(또는 파라미터만 추측하면) 이 데이터를 볼 수 있는가?" 답이 "그렇다"면, 그 데이터가 실제로 공개되어도 괜찮은 것인지 다시 판단하고, 아니라면 인증을 추가하세요.
이 3가지는 특히 LifeMap의 개인 분석결과(사주/성향/진로 리포트), ItemZone의 사업 아이디어/재무 정보처럼 "개인이 입력한 민감한 내용"을 다루는 기능에서 중요합니다. 설계 단계에서부터 반영해주시고, 구현 후에는 "이 API를 인증 없이 호출하면 무엇이 보이는가"를 한 번씩 직접 테스트해서 확인해주세요.
